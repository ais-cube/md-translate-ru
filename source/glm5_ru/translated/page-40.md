Пример данных
Каждый тестовый случай состоит из трёх компонентов: Задача, Чек-лист и
Выделенная среда. Ниже приведён типичный пример тестового случая:
Задача: Разработать онлайн-инструмент для рисования, включающий кисть, ластик, белый
холст и кнопку сохранения.
Цвет и толщина кисти должны выбираться с помощью кнопок слева.
Пользователи могут рисовать на холсте, нажимая и перетаскивая мышь.
Размер ластика должен выбираться с помощью кнопок слева. Пользователи могут
стирать содержимое, нажимая и перетаскивая мышь по холсту.
После завершения рисования нажатие кнопки «Сохранить» должно позволить
пользователю сохранить изображение локально.
Пожалуйста, реализуйте это с использованием фреймворка React в текущем каталоге.
Чек-лист:
Пользователь может выбрать цвет и толщину кисти с помощью кнопок слева,
рисование работает через нажатие и перетаскивание мыши на холсте.
Пользователь может выбрать размер ластика с помощью кнопок слева, стирание
работает через нажатие и перетаскивание мыши на холсте.
При нажатии кнопки «Сохранить» сгенерированное изображение успешно
сохраняется на локальной машине.
Построение и валидация данных
Мы реализуем строгий четырёхэтапный конвейер для обеспечения качества
данных:
• Этап 1: Синтез задач. Задачи разрабатываются старшими frontend-экспертами для обеспечения
отражения реальных инженерных задач при сохранении сбалансированного распределения по
различным сценариям и технологиям.
• Этап 2: Генерация и уточнение чек-листа. Мы сначала используем Claude Sonnet 4.5 для
синтеза чек-листов-кандидатов на основе спецификаций задач T. Затем они тщательно
проверяются и интегрируются экспертами. Посредством нескольких раундов уточнения мы
обеспечиваем, что каждый элемент проверки семантически однозначен, объективен и
обеспечивает исчерпывающее покрытие пользовательских требований.
• Этап 3: Коррекция на основе выполнения. Мы проводим перекрёстную валидацию между
фреймворком Agent-as-a-Judge и экспертами-людьми. Любые расхождения в оценке
запускают повторную оценку и исправление базовых данных для устранения потенциального
шума.
• Этап 4: Динамическая итерация бенчмарка. Для поддержания высокого уровня различительной
способности мы итеративно обновляем тестовый набор, удаляя тривиальные задачи, которые
больше не представляют сложности для современных агентов кодирования. Этот процесс
курирования экспертами завершился формированием финального набора из 220 высококачественных
frontend-задач по кодированию и соответствующих им чек-листов.
